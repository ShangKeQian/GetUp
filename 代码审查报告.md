## 一、高严重度问题

### H1. 竞态条件：`_toggle_detection` 在锁外替换检测器

**位置**：`main.py:86-119`（`_toggle_detection` 方法）

**问题**：
当用户快速"暂停→恢复"切换时，会产生检测器使用冲突。关键路径如下：

1. **暂停操作**：在锁内设置 `_running=False`、保存 `old_thread`、创建新 TimerEngine → **释放锁** → `old_thread.join(timeout=3)` → **在锁外**（第 116 行）`self._detector = PresenceDetector(...)`
2. **恢复操作**（在 join 期间发生）：获取锁 → `_running=True` → 启动新线程 → 新线程在 `_tick_loop` 第 122 行执行 `detector = self._detector`

如果"恢复"操作发生在"暂停"操作的 `join` 与检测器重建之间，新启动的 tick 线程会捕获**旧检测器**（即将被关闭的那个）。随后旧线程的 `finally` 块调用 `detector.close()`，而新线程仍在使用该已关闭的检测器——pynput 监听器已停止、摄像头已释放，导致静默失效（`check_once` 返回 `None`，`person_present` 恸为 `False`），用户不会被提醒。

**修复方案**：
```python
def _toggle_detection(self):
    old_thread = None
    with self._lock:
        if self._running:
            self._running = False
            self._tick_generation += 1
            old_thread = self._tick_thread
            self._timer = TimerEngine(
                work_minutes=self._config.work_minutes,
                break_minutes=self._config.break_minutes,
            )
            self._bind_timer_callbacks()
            self._tray._timer = self._timer
            self._tray.update_paused()
        else:
            # 检测器重建也放入锁内，确保新线程看到的是完整对象
            if old_thread:  # 之前有暂停操作待完成
                self._detector = PresenceDetector(
                    camera_index=self._config.camera_index,
                    sleep_timeout_minutes=self._config.sleep_timeout_minutes,
                )
            self._running = True
            self._tick_generation += 1
            self._tray.update_presence(True)
            self._tick_thread = threading.Thread(
                target=self._tick_loop, args=(self._tick_generation,), daemon=True
            )
            self._tick_thread.start()
            self._tray.update_running(self._running)

    if old_thread and old_thread.is_alive():
        old_thread.join(timeout=3)
    # 检测器重建移入锁内，此处不再需要
```

或者更彻底的方案：将"停止→等待旧线程→重建检测器"封装为独立方法，在锁内完成全部操作，`join` 使用 `Event` 信号替代超时轮询。

### H2. `_tick_loop` 无锁读取共享状态

**位置**：`main.py:121-148`

**问题**：
```python
def _tick_loop(self, generation):
    detector = self._detector          # 无锁读取，可能捕获正在被替换的对象
    timer = self._timer                # 无锁读取
    detector.start()

    while self._running and self._tick_generation == generation:  # 无锁读取
        ...
        if person_present != self._last_presence or sleeping != self._last_sleeping:
            self._last_presence = person_present     # 跨线程写
            self._last_sleeping = sleeping
```

- `self._detector`、`self._timer`、`self._running`、`self._tick_generation` 均在锁外读写
- `self._last_presence`、`self._last_sleeping` 由 tick 线程写入，由主线程在 `_update_main_window_status` 中读取，无同步保护

虽然 CPython 的 GIL 使单个变量读写原子化，但复合条件判断（如 `person_present != self._last_presence`）存在 TOCTOU 风险，且 `_detector` 可能在读取瞬间正被 `_toggle_detection` / `_restart_detection` 替换。

**修复方案**：
在 `_tick_loop` 入口处用锁快照 `detector` 和 `timer` 引用；对 `_last_presence` / `_last_sleeping` 使用 `threading.Lock` 或统一纳入已有的 `self._lock` 保护：
```python
def _tick_loop(self, generation):
    with self._lock:
        detector = self._detector
        timer = self._timer
    detector.start()
    while True:
        with self._lock:
            if not self._running or self._tick_generation != generation:
                break
        ...
```

### H3. `Config.save()` 无异常处理，写入失败导致崩溃

**位置**：`config.py:111-113`、`main_window.py:494`

**问题**：
```python
# config.py
def save(self):
    with open(self._path, "w", encoding="utf-8") as f:  # 无 try/except
        json.dump(self._data, f, indent=2, ensure_ascii=False)

# main_window.py
def _save(self):
    ...
    self._config.save()    # 若抛异常，后续代码不执行
    if self._on_save:
        self._on_save()    # 不会被调用
```

当配置文件路径不可写（只读属性、磁盘满、OneDrive 同步锁定文件）时，`save()` 抛出 `OSError` / `PermissionError`，`_save()` 未捕获，异常传播至 Qt 事件循环，应用可能崩溃或设置不会生效。用户无法得到任何反馈。

**修复方案**：
```python
# config.py
def save(self):
    try:
        with open(self._path, "w", encoding="utf-8") as f:
            json.dump(self._data, f, indent=2, ensure_ascii=False)
        return True
    except OSError as e:
        print(f"[Config] 保存失败: {e}", file=sys.stderr)
        return False

# main_window.py
def _save(self):
    ...
    if not self._config.save():
        self._save_btn.setText("保存失败 ✗")
        return
    if self._on_save:
        self._on_save()
```

### H5. 系统休眠/挂起后时间跳跃，计时器行为异常

**位置**：`timer.py:109-156`（`tick` 方法）

**问题**：
```python
dt = now - self._last_tick_time
self._last_tick_time = now
...
self._elapsed += dt          # 休眠后 dt 可能为数小时
...
self._break_remaining -= dt   # 休眠后立即变负数，遮罩秒关
```

当 Windows 进入睡眠/休眠时，`time.monotonic()` 在唤醒后会跳跃（`dt` 可能为数千秒）。后果：
- **TIMING 状态**：`_elapsed` 暴增，立即触发遮罩——即使用户刚回来
- **OVERLAY 状态**：`_break_remaining` 瞬间为负，遮罩立即关闭，休息时间被跳过

**修复方案**：
对 `dt` 设置上限，超过阈值视为系统挂起，重置计时器：
```python
dt = now - self._last_tick_time
self._last_tick_time = now
if dt > 60:  # 系统休眠/挂起
    self._elapsed = 0
    self._break_remaining = self._break_seconds
    return  # 跳过本次 tick
```

## 二、中严重度问题

### M1. `PresenceDetector.close()` 不持锁，存在并发使用风险

**位置**：`detectors/presence.py:86-94`

**问题**：
```python
def close(self):
    if self._closed:
        return
    self._closed = True
    if self._mouse_listener:
        self._mouse_listener.stop()
    if self._keyboard_listener:
        self._keyboard_listener.stop()
    self._camera.close()   # 不持 self._lock
```

`close()` 不获取 `self._lock`，而 `tick()` 在持锁状态下调用 `self._camera.check_once()`。如果 `close()` 在 `tick()` 执行期间被调用（例如 `_quit` 中 `join(timeout=3)` 超时后），摄像头在 `check_once` 过程中被释放/置 None，可能导致 `cv2` 底层崩溃或异常。

此外，`tick()` 不检查 `_closed` 标志，`close()` 后若 tick 线程仍存活（join 超时），会通过 `_ensure_open()` 重新打开摄像头，造成资源泄漏。

**修复方案**：
```python
def close(self):
    with self._lock:
        if self._closed:
            return
        self._closed = True
    # 在锁外停止监听器（stop 可能阻塞）
    if self._mouse_listener:
        self._mouse_listener.stop()
    if self._keyboard_listener:
        self._keyboard_listener.stop()
    self._camera.close()

def tick(self):
    with self._lock:
        if self._closed:
            return False
        ...
```

### M2. 摄像头检测在持锁状态下执行，阻塞 `wake()`

**位置**：`detectors/presence.py:35-75`

**问题**：
```python
def tick(self):
    with self._lock:                    # 持锁
        ...
        result = self._camera.check_once()  # 人脸检测，耗时 100-500ms
```

`check_once()` 涉及摄像头帧捕获 + MediaPipe 人脸推理，耗时可达数百毫秒。期间 `self._lock` 被持有，导致：
- `wake()` 阻塞（主线程 UI 卡顿）
- `is_sleeping` 虽然无锁读取但语义上可能不一致

**修复方案**：
将摄像头检测移出锁外，仅用锁保护状态字段的读写：
```python
def tick(self):
    now = time.time()
    with self._lock:
        idle_time = now - self._last_input_time
        need_camera_check = (not self._sleeping
                             and idle_time >= 5
                             and now - self._last_camera_found_time >= 5
                             and now - self._last_camera_check_time >= 5)
        if need_camera_check:
            self._last_camera_check_time = now

    # 锁外执行摄像头检测
    camera_result = None
    if need_camera_check:
        camera_result = self._camera.check_once()

    with self._lock:
        # 根据 camera_result 更新状态
        ...
```

### M3. `manual_break` 在锁外读取 `_break_remaining`

**位置**：`timer.py:95-107`

**问题**：
```python
def manual_break(self) -> bool:
    with self._lock:
        ...
        self._break_remaining = self._break_seconds
        self._overlay_paused = False
    # 锁已释放
    if self.on_show_overlay:
        self.on_show_overlay()
    if self.on_update_countdown:
        self.on_update_countdown(int(self._break_remaining))  # 无锁读取
```

`manual_break` 在锁外读取 `_break_remaining`。如果 tick 线程在此间隙执行了 `tick()` 并递减了 `_break_remaining`，传给 UI 的倒计时初始值会偏小。虽影响轻微（差约 1 秒），但违反了"共享状态锁内访问"的原则。

**修复方案**：
在锁内捕获需要传递的值：
```python
def manual_break(self) -> bool:
    with self._lock:
        if self._state != State.TIMING:
            return False
        self._state = State.OVERLAY
        self._break_remaining = self._break_seconds
        self._overlay_paused = False
        initial_countdown = int(self._break_remaining)
    if self.on_show_overlay:
        self.on_show_overlay()
    if self.on_update_countdown:
        self.on_update_countdown(initial_countdown)
    return True
```

### M5. 暂停监控时重置计时器，丢失工作进度

**位置**：`main.py:93-96`

**问题**：
```python
if self._running:
    self._running = False
    ...
    self._timer = TimerEngine(   # 创建全新引擎，elapsed 归零
```

```python
    old_thread = self._tick_thread
    # 不重建 timer，保留 elapsed
else:
    self._running = True
    self._tick_generation += 1
    # 复用现有 self._timer
    self._tick_thread = threading.Thread(...)
```

### M6. `_quit` / `_restart_detection` 在 join 超时后仍操作检测器

**位置**：`main.py:253-261`、`main.py:204-244`

**问题**：
```python
def _quit(self):
    ...
    if tick_thread and tick_thread.is_alive():
        tick_thread.join(timeout=3)      # 可能超时
    self._detector.close()                # 线程可能仍在使用检测器
    self._app.quit()
```

`_restart_detection` 第 230 行同理：`self._detector.close()` 在 join 后调用，但未检查线程是否真正退出。若摄像头 I/O 阻塞导致 join 超时，`close()` 在线程仍在运行时执行，触发 M1 所述的并发问题。

**修复方案**：
```python
if tick_thread and tick_thread.is_alive():
    tick_thread.join(timeout=3)
    if tick_thread.is_alive():
        print("[GetUp] tick 线程未在超时内退出", file=sys.stderr)
        # 不调用 close，让进程退出时 OS 回收
        self._app.quit()
        return
self._detector.close()
self._app.quit()
```

### M7. 配置文件存储在程序目录，重新打包后丢失

**位置**：`config.py:47-49`

**问题**：
```python
self._path = path or os.path.join(
```

```python
def _default_config_path():

        config_dir = os.path.join(base, APP_NAME)
        os.makedirs(config_dir, exist_ok=True)

    return os.path.join(os.path.dirname(os.path.abspath(sys.argv[0])), "config.json")
```

### L2. 时间源不统一

**位置**：`timer.py` 使用 `time.monotonic()`，`detectors/presence.py` 使用 `time.time()`

`time.time()` 可受系统时钟调整影响（NTP 校时、用户手动改时间），而 `time.monotonic()` 不会。建议统一使用 `time.monotonic()`，尤其 `_last_input_time` 的计算应不受时钟跳变影响。

### L4. `CameraDetector.check_once` 对持续读取失败无恢复机制

**位置**：`detectors/camera.py:38-48`

若 `cap.read()` 持续返回 `False`（摄像头被其他程序占用或硬件故障），`_ensure_open()` 认为摄像头已打开不会重置。建议引入连续失败计数器，超过阈值后 release 并重新打开：

```python
if not ret or frame is None:
```

### L5. `_CallbackSignal` 可能产生冗余的 `_drain` 调用

**位置**：`main.py:26-37`

多次 `post()` 会多次 `emit()`，队列中堆积多个 `_drain` 调用。第一个 `_drain` 清空队列后，后续 `_drain` 发现空队列直接返回。虽无功能问题，但有轻微开销。可引入 `_pending` 标志避免重复 emit。

---

### L6. `destroy_overlay` 未停止淡入动画

**位置**：`overlay.py:195-200`

若在淡入动画进行中调用 `destroy_overlay`，`_fade_anim` 仍在对已隐藏窗口操作 `windowOpacity`。虽无崩溃风险，但属资源浪费。建议：
```python
def destroy_overlay(self):
    if self._is_shown:
        self._is_shown = False
        if hasattr(self, '_fade_anim'):
            self._fade_anim.stop()
        self.hide()
        if self._on_close_callback:
            self._on_close_callback()
```

---

### L7. `CameraDetector.close()` 语义可被绕过

**位置**：`detectors/camera.py:56-63`

`close()` 将 `_face_detector` 置 None，但 `_ensure_open()` → `_init_face_detector()` 会重新创建。`close()` 并非终态，后续 `check_once` 仍可复活对象。命名易误导，建议将 `close()` 改名 `release_all()` 或在 `close()` 后设置 `_closed` 标志阻止复活。
